# 测试用例文档

## 📋 概述

本文档详细描述了AI驱动内容代理系统的测试用例，包括功能测试、性能测试、安全测试和用户体验测试的具体用例。每个测试用例都包含详细的测试步骤、预期结果和验收标准。

## 🎯 测试用例分类

### 1. 功能测试用例

#### 1.1 内容改写功能

**测试用例 TC-001: 基本内容改写**
- **测试目标**: 验证系统能够正确改写用户输入的内容
- **前置条件**: 用户已登录系统，API服务正常运行
- **测试步骤**:
  1. 导航到内容改写页面
  2. 在输入框中输入原始内容："这是一段测试内容，用于验证改写功能。"
  3. 选择改写风格："专业"
  4. 选择内容长度："中等"
  5. 点击"开始改写"按钮
  6. 等待处理完成
- **预期结果**:
  - 系统显示改写后的内容
  - 改写后内容与原始内容不同
  - 改写后内容保持原意
  - 显示字数统计和改写风格信息
- **验收标准**:
  - 改写成功率 ≥ 95%
  - 响应时间 ≤ 10秒
  - 内容质量评分 ≥ 4.0/5.0

**测试用例 TC-002: 不同风格改写**
- **测试目标**: 验证系统支持多种改写风格
- **前置条件**: 用户已登录系统
- **测试数据**:
  ```
  原始内容: "人工智能技术正在快速发展，对各行各业产生深远影响。"
  测试风格: ["专业", "通俗", "学术", "创意", "简洁"]
  ```
- **测试步骤**:
  1. 对每种风格执行改写操作
  2. 比较不同风格的改写结果
  3. 验证风格特征是否明显
- **预期结果**:
  - 不同风格的改写结果具有明显差异
  - 专业风格使用正式用词
  - 通俗风格使用日常用语
  - 学术风格包含专业术语
  - 创意风格表达新颖
  - 简洁风格内容精炼

**测试用例 TC-003: 长文本改写**
- **测试目标**: 验证系统处理长文本的能力
- **前置条件**: 用户已登录系统
- **测试数据**: 1000-5000字的长文本
- **测试步骤**:
  1. 输入长文本内容
  2. 选择改写参数
  3. 执行改写操作
  4. 验证改写结果的完整性
- **预期结果**:
  - 长文本能够完整改写
  - 保持原文结构和逻辑
  - 处理时间在合理范围内
- **验收标准**:
  - 支持最大5000字文本
  - 处理时间 ≤ 30秒
  - 内容完整性 ≥ 98%

  - 文章结构清晰，逻辑连贯
- **验收标准**:

  - 生成时间 ≤ 60秒

  - 关键词覆盖率 ≥ 80%

### 2. 性能测试用例

#### 2.1 响应时间测试

**测试用例 TC-301: API响应时间**
- **测试目标**: 验证API接口的响应时间
- **测试条件**: 正常负载下的单用户请求
- **测试接口**:
  - POST /api/content/rewrite
  - POST /api/content/generate
  - POST /api/render/{template}
- **测试步骤**:
  1. 发送标准测试请求
  2. 记录响应时间
  3. 重复测试100次
  4. 计算平均响应时间和P95值
- **性能指标**:
  - 内容改写：平均 ≤ 5秒，P95 ≤ 10秒
  - 文章生成：平均 ≤ 30秒，P95 ≤ 60秒
  - 模板渲染：平均 ≤ 2秒，P95 ≤ 5秒

**测试用例 TC-302: 并发性能测试**
- **测试目标**: 验证系统在并发请求下的性能
- **测试条件**: 模拟多用户并发访问
- **测试场景**:
  - 10个并发用户
  - 50个并发用户
  - 100个并发用户
- **测试步骤**:
  1. 使用负载测试工具模拟并发请求
  2. 监控系统资源使用情况
  3. 记录响应时间和错误率
  4. 分析性能瓶颈
- **性能指标**:
  - 10并发：响应时间增长 ≤ 20%
  - 50并发：响应时间增长 ≤ 50%
  - 100并发：响应时间增长 ≤ 100%
  - 错误率 ≤ 1%

#### 2.2 资源使用测试

**测试用例 TC-303: 内存使用测试**
- **测试目标**: 验证系统内存使用情况
- **测试条件**: 长时间运行和高负载场景
- **测试步骤**:
  1. 监控系统启动时的内存使用
  2. 执行大量请求操作
  3. 监控内存使用变化
  4. 检查是否存在内存泄漏

**测试用例 TC-304: CPU使用测试**
- **测试目标**: 验证系统CPU使用效率
- **测试条件**: 不同负载级别下的CPU使用
- **测试步骤**:
  1. 监控空闲状态CPU使用率
  2. 执行单个请求时的CPU使用
  3. 执行并发请求时的CPU使用
  4. 分析CPU使用模式

  - 单请求CPU峰值 ≤ 50%
  - 并发请求CPU平均 ≤ 80%

### 3. 安全测试用例

#### 3.1 输入验证测试

**测试用例 TC-401: XSS攻击防护**
- **测试目标**: 验证系统对XSS攻击的防护能力
- **测试条件**: 各种XSS攻击载荷
- **测试数据**:
  ```javascript
  const xssPayloads = [
    '<script>alert("XSS")</script>',
    '<img src="x" onerror="alert(1)">',
    'javascript:alert("XSS")',
    '<iframe src="javascript:alert(1)"></iframe>',
    '<svg onload="alert(1)">',
    '" onmouseover="alert(1)"',
    '<body onload="alert(1)">'
  ];
  ```
- **测试步骤**:
  1. 在各输入字段中输入XSS载荷
  2. 提交请求并观察响应
  3. 检查输出是否被正确转义
  4. 验证脚本是否被执行
- **预期结果**:
  - 所有XSS载荷被正确过滤或转义
  - 恶意脚本不会在浏览器中执行
  - 系统返回安全的内容

**测试用例 TC-402: SQL注入防护**
- **测试目标**: 验证系统对SQL注入攻击的防护
- **测试条件**: 各种SQL注入载荷
- **测试数据**:
  ```sql
  const sqlPayloads = [
    "'; DROP TABLE users; --",
    "1' OR '1'='1",
    "UNION SELECT * FROM admin",
    "'; INSERT INTO users VALUES ('hacker', 'password'); --",
    "1' AND (SELECT COUNT(*) FROM users) > 0 --"
  ];
  ```
- **测试步骤**:
  1. 在输入参数中注入SQL载荷
  2. 监控数据库查询日志
  3. 检查是否执行了恶意SQL
  4. 验证数据完整性
- **预期结果**:
  - SQL注入载荷被正确过滤
  - 数据库查询使用参数化查询
  - 数据库结构和数据未被破坏

#### 3.2 认证授权测试

**测试用例 TC-403: API密钥验证**
- **测试目标**: 验证API密钥的验证机制
- **测试条件**: 有效和无效的API密钥
- **测试步骤**:
  1. 使用有效API密钥访问接口
  2. 使用无效API密钥访问接口
  3. 使用过期API密钥访问接口
  4. 不提供API密钥访问接口
- **预期结果**:
  - 有效密钥：正常访问
  - 无效密钥：返回401错误
  - 过期密钥：返回401错误
  - 无密钥：返回401错误

**测试用例 TC-404: 权限控制测试**
- **测试目标**: 验证用户权限控制机制
- **测试条件**: 不同权限级别的用户
- **测试步骤**:
  1. 普通用户访问管理接口
  2. 管理员用户访问管理接口
  3. 测试权限边界情况
  4. 验证权限提升攻击防护
- **预期结果**:
  - 普通用户无法访问管理接口
  - 管理员用户可以正常访问
  - 权限检查严格执行
  - 防止权限提升攻击
